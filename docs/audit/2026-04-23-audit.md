## Findings

### FINDING-001 — Terminal escapes from file contents pass to the TTY — **HIGH**

**Location:** `src/main.cyr` — `render_chunk` (paths 1/2/3) and
`render_highlighted_buf`.

**Issue.** In decorated or highlighted mode owl writes the file's
content bytes verbatim to stdout. A file containing ANSI / OSC / DCS
control sequences is interpreted by the user's terminal the same way
the escapes emitted by owl itself are. Weaponized vectors are well
documented in the class:

- **OSC 52** — `ESC ] 52 ; c ; <base64> BEL` silently writes to the
  system clipboard. xterm, kitty, foot, WezTerm, Alacritty, iTerm2
  honor it.
- **OSC 0/1/2 + CSI `t` title report** — set the window title to a
  fake shell prompt, then request the terminal report its title back
  as keystrokes onto stdin. The canonical "cat considered harmful"
  RCE, alive since xterm CVE-2003-0063.
- **CSI `c` / `CSI Ps n`** — Device Attributes / Device Status
  Report; reply bytes land on stdin and execute at next Enter.
- **OSC 1337 (iTerm2)** — CVE-2019-9535 (iTerm2 ≤ 3.3.5, High) was
  the most recent RCE in this family, via tmux-integration OSCs
  embedded in file content.
- **OSC 8** — hyperlinks; not RCE on their own, but CVE-2024-32487
  (`less` before 606) showed that even "strip all but SGR" filters
  get OSC 8 wrong by default.

**Risk to owl.** owl's v1 target is "drop-in `cat`/`bat` replacement",
which means users will point it at logs, network captures, and
adversarial content. The class floor is "viewer must not let the
file drive the terminal."

**Recommended fix.** In decorated, highlighted, or colored output
modes, emit only SGR (`CSI … m`) from the *file* bytestream and
strip (or visibly render as `^[[…`) every other escape. owl already
owns the rendering pipeline — vyakarana tokens are colored by owl's
own `ansi_fg` / `ansi_reset`, so the SGR escapes we emit are
trusted; everything else from `load8(buf+i)` must not be.

Add `--raw-control-chars` / `-r` (mirroring `less -r`) as the
explicit opt-out. `-p` (plain / cat-parity) must still pass
everything through — that's the literal cat contract.

**Effort.** Medium. A byte classifier in `render_chunk` and
`render_highlighted_buf`: when `c == 0x1B`, scan to end of the
escape (ESC `[`…final, ESC `]`…BEL/ST, ESC `P`/`_`/`^`/`X`…ST),
allow-list SGR, replace everything else with `^[` + visible
representation.

### FINDING-002 — Control bytes in echoed paths reach stderr — **MEDIUM**

**Location:**
- `src/main.cyr:202` — `report_error(path, reason)` emits `path`
  via `eprint(path, strlen(path))`
- `src/main.cyr:632` — large-file notice inline
- `src/config.cyr:121` — `_cfg_err(path, line, reason)`

**Issue.** Paths come from argv (user shell), argv-substituted
environment variables (`$OWL_CONFIG`), and filesystem walks. Any of
them can contain `0x1B` and other C0 control bytes. owl echoes them
to stderr uncensored. A crafted path like `$'\x1b]0;evil\x07'` will
set the user's terminal title; `$'\x1b[2J'` clears the screen; an
OSC-52 payload writes the clipboard.

This is the same class vyakarana's FINDING-006 (2026-04-23) closed
with a `_sanitize_for_stderr` helper: replace bytes < 0x20 and
0x7F (DEL) with `?`; let UTF-8 (≥ 0x80) pass. owl inherits the
hazard from the same echo-paths-to-stderr pattern.

**Recommended fix.** Port the vyakarana helper. Touch points:
`report_error`, the large-file inline writer, `_cfg_err`. UTF-8
paths still render legibly; controls become safe.

**Effort.** Small. ~20 lines, one helper, three call-site edits.

### FINDING-003 — `git diff` invoked via `/bin/sh -c` — **MEDIUM**

**Location:** `src/vcs.cyr` — `_build_git_cmd` (line 73) and
`_capture_shell` (line 95).

**Issue.** The current invocation:

```
/bin/sh -c "git diff --no-color -U0 -- '<path>' 2>/dev/null"
```

is defended by (a) single-quote wrapping of the path and (b) a
reject-list of `'`, ESC, bytes < 0x20, and DEL. That defends
against classical shell injection **today** but the model —
serialize argv into a string, then blocklist dangerous bytes — is
the anti-pattern flagged by every modern security guide. Failure
modes:

- Any future change that appends a user-controlled flag (e.g.
  honoring a `vcs_context = 3` config to add `-U3`) is one
  interpolation typo away from a CVE.
- The reject-list doesn't cover high-bit sequences that a locale
  or future shell could treat specially.
- Comparable tools (`bat`, `delta`, `difftastic`, `fzf`, modern
  `git` plumbing) have all moved off `sh -c`. `less` had to cut
  CVE-2022-46663 (shell-metachar in filenames via `LESSOPEN`)
  for precisely this pattern.

**Recommended fix.** Move the invocation to `execve` with argv:

```
execve("/usr/bin/git",
       ["git", "-C", <dirname>, "diff", "--no-color", "-U0",
        "--", <basename>], envp)
```

Resolve `git`'s absolute path either by trying a short list
(`/usr/local/bin/git` → `/usr/bin/git` → `/bin/git`) or by
shipping a `which`-style helper. The existing pipe/fork/capture
code stays; only the exec call changes.

`--` before the path keeps CVE-2017-1000117-style
option-injection closed (git already treats everything after `--`
as positional).

**Effort.** Small. ~30 lines in `vcs.cyr`. Removes the need for
`_path_shell_safe` reject-list entirely — argv boundaries are
kernel-enforced.

**Bonus.** When SIT ships, this also becomes the shape the SIT
rewrite replaces, so we're not throwing work away.

### FINDING-004 — `waitpid` writes 4 bytes into `var buf[1]` — **LOW**

**Location:**
- `src/pager.cyr:97` — `var status_buf[1]; sys_waitpid(…, &status_buf, …);`
- `src/vcs.cyr:144` — `var stbuf[1]; sys_waitpid(…, &stbuf, …);`

**Issue.** Per CLAUDE.md, `var buf[N]` is N bytes. `sys_waitpid`
writes a kernel `int` — 4 bytes — into the supplied pointer.
Declaring `var status_buf[1]` and asking the kernel to populate
it is a 3-byte overrun of whatever the bump allocator placed
next in memory. The Cyrius stdlib's own `process.cyr` has the
same pattern, so this is an established convention that has been
working by luck (likely 8-byte alignment in the bump arena). It
still violates the explicit `var buf[N] = N bytes` contract.

**Risk.** No exploit path in practice today — the overrun lands
on alignment padding. An allocator or codegen change could
silently introduce memory corruption.

**Recommended fix.** Declare `var status_buf[8]` everywhere
`waitpid`'s status pointer is passed. One-line change at each
site. Optionally file an upstream cleanup against Cyrius stdlib.

**Effort.** Trivial.

### FINDING-005 — Paths containing `'` silently render without VCS markers — **LOW**

**Location:** `src/vcs.cyr:52` — `_path_shell_safe` returns 0
for paths containing `'`, `< 0x20`, or `0x7F`.

**Issue.** The user sees a file that has VCS changes but owl's
gutter shows none, with no indication why.

**Status.** Superseded by the fix for FINDING-003 — moving to
`execve` removes the need to reject `'` at all. No standalone
fix needed if 003 lands.

### FINDING-006 — Diff output capped at 64 KB — **INFO**

**Location:** `src/vcs.cyr:268` — `diff_cap = 65536`.

**Issue.** For a file with more than 64 KB of unified-diff
output, markers are partial. Not a security issue; the gutter
simply under-reports for huge diffs.

**Status.** Documented as a known M6 scaffold limit. Lifts when
SIT swap lands (no shell-out, direct library call).

---

### FINDING-007 — stdout write returns not checked — **INFO**

**Location:** every `syscall(1, 1, …)` and `file_write(STDOUT, …)`
call site.

**Issue.** Design-spec §9 defines exit code 3 for "I/O error on
stdout". owl never emits 3. Cleanly exits 0 on SIGPIPE (FINDING-008)
because the default signal-handler terminates before the write
returns; other stdout errors (disk full on `>`, device unplug) are
silently swallowed.

**Status.** Acceptable for v1 per the §11 note ("may also be
silently suppressed"). Revisit in post-v1 if real users hit it.

### FINDING-008 — SIGPIPE posture verified clean — **INFORMATIONAL**

**Verification.**

```
$ yes line | head -100000 > /tmp/big.txt
$ ./build/owl /tmp/big.txt | head -1 > /tmp/out 2> /tmp/err
$ echo "exit=$? stdout=$(wc -c</tmp/out) stderr=$(wc -c</tmp/err)"
exit=0 stdout=5 stderr=0
```

No "Broken pipe" leak, no non-zero exit. The Cyrius runtime is
not masking SIGPIPE — owl's write returns the SIGPIPE to the
kernel which terminates cleanly. This is the correct POSIX
posture; `bat` and `ripgrep` had to explicitly restore
`SIG_DFL` at startup to get the same behavior out of Rust.

## External research — class context

**`less`** recently shipped two relevant CVEs:

- **CVE-2022-46663** (Dec 2022, High, `less` ≤ 590) —
  shell-metachar filenames via `LESSOPEN` callback. Fix: sanitize
  filenames before argv dispatch. Lesson: if owl ever adds a
  preprocessor hook, exec via argv, not `sh -c` (applies to
  FINDING-003 today).
- **CVE-2024-32487** (Apr 2024, High, `less` before 606) — OSC 8
  hyperlinks weren't properly stripped in `-R` mode. Lesson:
  "allow SGR, strip rest" is subtle; a byte-level parser must
  understand ESC `]` … ST/BEL, ESC `P` … ST, etc. (informs
  FINDING-001's classifier shape).

**`bat`** has no CVE record against it directly but:

- Uses `libgit2` in-process for the git gutter rather than
  shelling out to `git`. FINDING-003 chases the same posture.
- Does not ship `--strip-ansi`; there are tracked issues asking
  for it. owl shipping that default would be a win over the
  comparator.
- Does binary-file detection (null-byte sniff) — owl shipped
  equivalent behavior in M8a. Parity achieved.

**Git / rg** defaults:

- `git` since 2.9 sanitizes its own output paths (e.g. in
  `log --format`) for control bytes. `rg` does the same on error
  paths. FINDING-002 asks owl to do the equivalent.

**Terminal-escape triage** (which classes to strip by default
in decorated mode, per the research brief):

| Sequence                        | Risk                         | Strip in decorated? |
|---------------------------------|------------------------------|---------------------|
| CSI … `m` (SGR color)           | Benign                       | Allow               |
| CSI … `K` (erase line)          | Benign-ish                   | Allow               |
| CSI … `J` (erase display)       | Annoying                     | Strip               |
| CSI … `H` / `f` (cursor pos)    | Overwrite prompt             | Strip               |
| CSI … `t` (window ops)          | Title report → RCE           | **Strip**           |
| CSI … `c` / `n` (DA / DSR)      | Reply-to-stdin → RCE         | **Strip**           |
| OSC 0/1/2 (title)               | Spoof + title-report combo   | **Strip**           |
| OSC 8 (hyperlink)               | Clickjacking                 | Strip (opt-in)      |
| OSC 52 (clipboard)              | Silent clipboard write       | **Strip**           |
| OSC 133 (shell integration)     | Prompt spoofing              | Strip               |
| OSC 1337 (iTerm2)               | Historical RCE (CVE-2019-9535) | **Strip**           |
| DCS / APC / PM / SOS            | Parser confusion             | **Strip**           |
| C1 (0x80–0x9F)                  | Often alias for ESC + char   | **Strip**           |

## Recommended landing order for fixes

1. **FINDING-002** (path sanitization) — smallest patch, closes
   a real stderr-capture leak, mirrors a fix we already wrote in
   vyakarana. Land in the next owl patch.
2. **FINDING-004** (waitpid buffer size) — trivial, latent hazard.
   Land in the same patch as 002.
3. **FINDING-003** (argv for git diff) — medium patch, removes an
   entire class (shell injection) and implicitly resolves
   FINDING-005. Target the M8c docs pass cycle.
4. **FINDING-001** (ANSI passthrough) — the largest behavior
   change. Introduce `--raw-control-chars` flag and a byte
   classifier in the render loops. Target M8c; ships as a
   documented flag in the first release candidate.

Findings 006–008 are tracked in ROADMAP but require no code.

## Closeout

owl's attack surface is small and well-contained:

- **Buffer safety** — every `alloc(N)` and `var buf[N]` was
  walked; all bounded correctly except FINDING-004.
- **Syscall usage** — every `syscall(…)` goes through the
  stdlib wrappers or is a short-list (`SYS_IOCTL` for TTY,
  `SYS_WRITE` / `SYS_DUP` / `SYS_EXIT`). Returns are checked
  where divergence matters.
- **Shell invocation** — two sites (`pager.cyr`, `vcs.cyr`).
  Pager is trusted (user env). VCS is FINDING-003.
- **Input validation** — argv paths validated before shell
  interpolation; config values validated via the same
  `set_*` helpers the CLI uses; grammar file names come from
  a hardcoded allow-list.
- **Integer / pointer safety** — all i64, all bounded.

No CRITICAL findings. Four OPEN findings totaling ~200 lines of
patch work to close the full set.

### FINDING-002 closed

- New helper `eprint_sanitized(s, n)` in `src/main.cyr` — replaces
  bytes `< 0x20` and `0x7F` with `'?'`, passes UTF-8 (`>= 0x80`)
  through.
- Wired into `report_error`, the large-file fallback notice, and
  `_cfg_err` in `src/config.cyr` — every stderr path that echoes
  user-supplied strings now passes through the filter.
- Smoke gate: an ESC-in-path probe ($'\x1b]0;evil\x07badpath')
  confirms no raw ESC reaches stderr.

### FINDING-003 and FINDING-005 closed

- Removed `_path_shell_safe`, `_build_git_cmd`, `_capture_shell`
  from `src/vcs.cyr`.
- New `_find_git()` tries `/usr/bin/git` → `/usr/local/bin/git` →
  `/bin/git` via `sys_open(path, O_RDONLY, 0)` — no PATH lookup,
  no shell involvement.
- New `_run_git_diff(path, buf, cap)` forks and `sys_execve`s with
  explicit argv: `["git", "diff", "--no-color", "-U0", "--", path]`.
  Child closes stderr to `/dev/null` so "not a git repository" etc.
  never reach owl's stderr. Kernel enforces argv boundaries — no
  quoting bugs possible; paths containing `'`, `$`, spaces, or
  anything else pass through unchanged (hence 005 closes for
  free).
- Smoke gate: argv-git still produces `+` markers on a dirty
  README.md inside the repo.

### FINDING-001 closed

- New state machine in `src/main.cyr`: `g_esc_state` (5 states —
  normal, saw-ESC, in-CSI, in-string, in-string-saw-ESC) plus
  `_emit_file_byte(c)` classifier that returns 0 when the byte
  is part of a file-origin escape.
- Integrated into both `render_chunk` (fast paths gated behind
  `strip == 0`; path 3 filters content bytes) and
  `render_highlighted_buf` (same filter, after the gutter).
- New global `g_flag_raw_controls` + `-r` / `--raw-control-chars`
  CLI flag as the explicit opt-out, mirroring `less -r`.
- Gate: `_strip_active()` is true when `g_want_color == 1` and
  none of `-p`, `-A`, or `--raw-control-chars` are in effect.
- `render_reset()` now clears `g_esc_state` per file.
- Smoke gate: a file containing an OSC 52 clipboard-write sequence
  renders as `beforeafter` by default; `-r` / `-p` pass it through;
  `-A` renders it visibly as `^[]52;…^G`.

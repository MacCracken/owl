# ADR 0008: 1.1.1 release scope and pacing

- **Status**: Accepted
- **Date**: 2026-04-25
- **Deciders**: @MacCracken

## Context

After 1.1.0 (stdin highlight fix), the 1.x backlog had a cluster of
S-effort CLI ergonomics improvements: `--version --verbose`,
`--strip-ansi=*`, `--line-range=A:B`, per-language config overrides
(`ext.<extension>`), and `--wrap=character`. Each is contained, has
a smoke gate, and adds no new subsystems.

Strict SemVer would put new flags in **minor** versions (1.2.0,
1.3.0, …). owl's pre-2.0 phase is short, and the cost of a
1.2.0 / 1.3.0 / 1.4.0 / 1.5.0 / 1.6.0 sequence — each carrying
one small feature — is friction for both users (more bumps to
track) and the project (changelog churn, version-bump bookkeeping).

The opposite extreme — wait for one large minor — means users
don't get incremental value as features land.

## Decision

We will treat owl's pre-2.0 patch versions as "small, frequent
ergonomics drops." 1.1.1 ships all five S-tier features as a
single patch release. The CHANGELOG distinguishes Added / Fixed /
Changed sections so consumers can audit what they're picking up.
Each feature has its own smoke gate; the release is gated on full
test + smoke + lint passing from a clean checkout.

Larger structural changes (new dependencies, breaking flags, new
subsystems) will continue to use minor or major bumps as
appropriate.

## Consequences

- 1.1.1 ships ~354 lines of source + ~7 KB binary growth across
  five features.
- Users can audit changes via the per-feature CHANGELOG entries;
  no one lands without a smoke gate.
- This ADR sets precedent: 1.1.2 (content fallbacks: hex-dump,
  user grammars/themes) and 1.1.3 (smarter detection + `--diff`)
  may bundle multiple features each, again as patches.
- If a user reads strict SemVer into the version string and gets
  surprised by a new flag in a patch, the README and CHANGELOG
  set the correct expectation.
- Revisited at 2.0.0 cut: post-2.0 may shift to stricter SemVer.
